# 历史弱点扫描器

> "历史是最好的批评家。他们以为是新发明，我翻出来，原来三十年前就有人失败过。"
> —— 李敖

## 核心任务

**用历史当镜子，照出当前论断的本质。** 工程领域很少有真正全新的东西。大多数"新技术"是旧思路在新包装下的重演。历史弱点扫描的任务是：找出这个东西的历史原型，查清楚它们为什么失败，然后检验当前版本有没有真的解决那些失败的原因。

---

## 扫描步骤

### 第一步：寻找历史原型

这个技术/方案的历史上有没有结构相似的前辈？

**搜索方向：**
- 同名或近名的历史项目（直接换皮）
- 解决相同问题的不同名称方案
- 同一思路在不同领域的历史应用

**各工程领域的历史原型库（部分）：**

**计算机工程：**
- SOA → 微服务（同一思路，不同粒度）
- 大型机集中计算 → 云计算（集中化的轮回）
- 对象数据库 → 文档数据库（非关系型的重演）
- 专家系统 → 现代AI决策系统（规则引擎的变体）
- J2EE企业应用 → 当代微服务生态（复杂度的转移）

**建筑/土木：**
- 预制装配式建筑（1960年代苏联式建筑的重演）
- 智慧城市（每十年一次的新概念）
- 轻钢结构住宅（多次兴起又沉寂）

**机械/自动化：**
- 刚性自动化 → 柔性制造 → 工业4.0（反复摆动）
- 液压控制 → 电控 → 混合控制（各有适用场景）

**产品设计：**
- 模块化设计（每个时代都被重新发现）
- 可持续设计（更换标签的耐久性设计）

### 第二步：分析历史失败原因

找到历史原型后，查清楚它们失败或受限的真正原因：

**失败原因分类：**

| 原因类型 | 含义 | 检验当前版本是否解决 |
|---------|------|------------------|
| **时代约束** | 当时缺少某种技术/材料/基础设施 | 当前这个约束是否已消除？ |
| **成本约束** | 当时成本无法支撑大规模推广 | 成本曲线是否真的已经改变？ |
| **组织约束** | 需要组织能力当时大多数团队不具备 | 今天的平均组织能力是否已足够？ |
| **本质缺陷** | 技术本身有不可克服的结构性问题 | 当前版本有没有改变这个结构？ |
| **市场时机** | 技术对了但市场没准备好 | 现在市场准备好了吗？ |

### 第三步：检验当前版本的真实进步

这个版本宣称比历史版本进步在哪里？

**检验方法：**
1. 列出宣称的改进点
2. 逐一检验：这个改进是否真的解决了历史失败的原因？
3. 还是只是换了名字和界面，核心问题依然存在？

**常见的伪进步模式：**
- 用新工具重新实现旧方案（问题没变，工具变了）
- 解决了历史失败的次要原因，主要原因还在
- 把失败的约束条件标注为"不在此次讨论范围"

### 第四步：评估历史教训的适用性

历史教训有时确实已经过时。公正评估：

- **技术环境的变化**：某些历史失败确实是因为当时技术不成熟，现在已经改变
- **规模的变化**：某些历史失败在小规模时不明显，当前规模下才会触发
- **生态的变化**：某些历史失败是因为缺少生态支撑，现在生态已成熟

**结论标准：**
- 历史教训仍然完全适用：当前版本没有真正解决历史失败原因
- 历史教训部分适用：解决了部分原因，但核心问题仍在
- 历史教训已过时：有充分证据说明关键约束已改变

```

历史原型：
  - [原型名称]：[时期] [相似点]
  - [原型名称]：[时期] [相似点]

历史失败原因：
  - [原因1]：[类型] [当时具体情况]
  - [原因2]：[类型] [当时具体情况]

当前版本的宣称改进：
  - [改进1]：[是否真正解决历史问题] [理由]
  - [改进2]：[是否真正解决历史问题] [理由]

历史教训适用性评估：
  [完全适用 / 部分适用 / 已过时] + 理由

最有力的历史证据：
[一个最能说明这个"新东西"本质上是旧问题的具体历史案例]
```

## 向下游传递

- `attack-writer`（攻击文）：历史先例是征引武器的重要来源——"这事以前就失败过"比任何逻辑论证都有力
- `other-mountains`（他山之石）：历史失败原因的分类（时代约束、本质缺陷等）可直接映射为跨领域解法的搜索关键词
