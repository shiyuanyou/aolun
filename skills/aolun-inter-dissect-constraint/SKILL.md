# 约束层解剖器

> "任何东西都有它的适用范围。把适用范围藏起来卖给你，这叫欺诈。"
> —— 李敖

## 核心任务

**找到边界，然后把边界放大。** 没有任何技术在所有条件下都成立。约束层解剖的任务是：找出这个技术真正成立的前提条件，然后检验这些条件在现实中有多难满足。

---

## 解剖步骤

### 第一步：建立约束清单

从四个维度系统地列出约束：

**规模约束**
- 这个技术在什么量级下被验证过？（数据量、并发量、结构尺寸、生产批量……）
- 从验证量级扩展到目标量级，会发生什么？
- 是否存在规模拐点（超过某个量级，性质改变）？

**环境约束**
- 这个技术依赖什么环境条件？（温度、湿度、网络质量、电源稳定性……）
- 实验室/演示条件 vs. 真实部署条件的差距是多少？
- 极端条件下（故障、高峰、老化）的行为是什么？

**依赖约束**
- 这个技术依赖哪些外部系统/组件/标准？
- 如果依赖项不稳定或消失，这个技术还能独立存在吗？
- 依赖项的版本锁定问题有多严重？

**人力与组织约束**
- 实施这个技术需要什么技能水平的人？
- 需要多大的组织协调成本？
- 运维和长期维护的真实代价是什么？

### 第二步：检验约束的被披露程度

| 约束类型 | 帖子是否提及？ | 是否轻描淡写？ |
|---------|-------------|------------|
| 规模约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 环境约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 依赖约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 人力约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |

**刻意隐藏约束**是比技术本身有缺陷更严重的问题。

### 第三步：测试边界条件

对每个关键约束，做边界压力测试：

```
约束：[X 必须满足条件 C]
测试问题：如果 C 只满足 80%？50%？完全不满足？
实际情况下 C 的满足概率是多少？
```

**各工程领域的典型边界压力点：**

- **计算机工程**：流量突增 10x、网络分区、磁盘满载、依赖服务宕机
- **建筑/土木**：地震设防烈度、50年一遇洪水、地基差异沉降
- **机械工程**：超载 150%、温度循环 1000 次、润滑失效
- **自动化/控制**：传感器失效、通信中断 30 秒、执行器饱和
- **产品设计**：跌落 1.5m、温度 -20°C 到 60°C 循环、10 年使用磨损

### 第四步：估算约束违反的代价

约束被违反后：
- 系统是**优雅降级**（性能下降但不崩溃）还是**灾难性失效**？
- 失效的**可见性**如何？（能不能被发现？多久能被发现？）
- **恢复成本**是多少？

## fileflow 模式说明

当由 aolun-fileflow 编排执行时，本 skill 的输出和引用遵循以下规则：

1. **输出格式**：文件顶部必须包含【摘要】块（3-5行，给出本步骤核心判断），然后是 --- 分隔线，然后是完整分析内容。

2. **引用硬约束**：每个弱点的证据必须附原文定位和直接引用。
   - 粘贴/单文件模式：引用格式为 `第<N>行："直接引用原句"`
   - 目录模式：引用格式为 `<文件名>:<N>："直接引用原句"`
   - 连续引用超过 3 行时，用行号范围标注：`第<N>-<M>行` 或 `<文件名>:<N>-<M>："..."`
   - 禁止转述代替引用——转述必然失真，引用才是弹药

3. **证据搜索**：使用 Read 和 Grep 工具主动搜索原文，不得等待主 agent 传递文本。
   - 粘贴/单文件模式：搜索 `00-original.md` 或源文件路径
   - 目录模式：搜索 `00-sources/` 下的 .md 文件（从 `00-index.md` 获取文件清单，或搜索源目录下 .md 文件）
   - 搜索策略优先使用 `00-todolist.md` 中"语义线索"部分的精确行号

4. **向下游传递**：在 fileflow 模式下，本 skill 的输出自动写入编号文件，下游 skill 通过文件路径读取。本 skill 无需关心下游传递机制。
