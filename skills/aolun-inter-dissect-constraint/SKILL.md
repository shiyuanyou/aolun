# 约束层解剖器

> "任何东西都有它的适用范围。把适用范围藏起来卖给你，这叫欺诈。"
> —— 李敖

## 核心任务

**找到边界，然后把边界放大。** 没有任何技术在所有条件下都成立。约束层解剖的任务是：找出这个技术真正成立的前提条件，然后检验这些条件在现实中有多难满足。

---

## 解剖步骤

### 第一步：建立约束清单

从四个维度系统地列出约束：

**规模约束**
- 这个技术在什么量级下被验证过？（数据量、并发量、结构尺寸、生产批量……）
- 从验证量级扩展到目标量级，会发生什么？
- 是否存在规模拐点（超过某个量级，性质改变）？

**环境约束**
- 这个技术依赖什么环境条件？（温度、湿度、网络质量、电源稳定性……）
- 实验室/演示条件 vs. 真实部署条件的差距是多少？
- 极端条件下（故障、高峰、老化）的行为是什么？

**依赖约束**
- 这个技术依赖哪些外部系统/组件/标准？
- 如果依赖项不稳定或消失，这个技术还能独立存在吗？
- 依赖项的版本锁定问题有多严重？

**人力与组织约束**
- 实施这个技术需要什么技能水平的人？
- 需要多大的组织协调成本？
- 运维和长期维护的真实代价是什么？

### 第二步：检验约束的被披露程度

| 约束类型 | 帖子是否提及？ | 是否轻描淡写？ |
|---------|-------------|------------|
| 规模约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 环境约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 依赖约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |
| 人力约束 | ☐ 是 ☐ 否 | ☐ 是 ☐ 否 |

**刻意隐藏约束**是比技术本身有缺陷更严重的问题。

### 第三步：测试边界条件

对每个关键约束，做边界压力测试：

```
约束：[X 必须满足条件 C]
测试问题：如果 C 只满足 80%？50%？完全不满足？
实际情况下 C 的满足概率是多少？
```

**各工程领域的典型边界压力点：**

- **计算机工程**：流量突增 10x、网络分区、磁盘满载、依赖服务宕机
- **建筑/土木**：地震设防烈度、50年一遇洪水、地基差异沉降
- **机械工程**：超载 150%、温度循环 1000 次、润滑失效
- **自动化/控制**：传感器失效、通信中断 30 秒、执行器饱和
- **产品设计**：跌落 1.5m、温度 -20°C 到 60°C 循环、10 年使用磨损

### 第四步：估算约束违反的代价

约束被违反后：
- 系统是**优雅降级**（性能下降但不崩溃）还是**灾难性失效**？
- 失效的**可见性**如何？（能不能被发现？多久能被发现？）
- **恢复成本**是多少？

```
【约束层解剖报告】

约束清单：
规模约束：
  - [约束1]：验证边界 [X]，目标规模 [Y]，差距 [Z]
环境约束：
  - [约束2]：实验室条件 [A]，真实条件 [B]，差距 [C]
依赖约束：
  - [约束3]：依赖 [D]，若失效：[后果]
人力约束：
  - [约束4]：需要 [E]，现实可得性 [F]

披露检验：
  - 哪些约束被刻意隐藏或轻描淡写：[列表]

边界压力测试结果：
  - 最脆弱的边界：[约束X]，触发条件：[Y]，失效模式：[Z]

约束层综合判断：
[这个技术在什么条件下有用，什么条件下是骗局]
```
