# 逻辑弱点扫描器

> "逻辑不对，事实再多也是废话。逻辑对了，一个事实就够了。"
> —— 李敖

## 核心任务

**把论证的骨架暴露出来，然后看骨架哪里断了。** 技术帖子的错误往往不是数据造假，而是推论跳跃——数据是真的，但结论从数据里跳不出来。

---

## 扫描清单

### 一、推论有效性检验

把帖子的核心论证还原成标准形式：

```
前提1：______
前提2：______
∴ 结论：______
```

**检验问题：**
- 结论是否逻辑上必然从前提导出？
- 还是只是"可能"或"大概率"导出？
- 如果前提都为真，结论还有可能为假吗？

### 二、常见逻辑谬误扫描

| 谬误类型 | 识别特征 | 工程领域示例 |
|---------|---------|------------|
| **后此谬误** | "用了A之后B发生了，所以A导致B" | "上线新架构后性能提升了，架构是原因" |
| **幸存者偏差** | 只统计成功案例 | "所有独角兽都用了微服务" |
| **错误类比** | 两个表面相似但结构不同的事物 | "软件架构就像建筑，所以要先设计图纸" |
| **滑坡谬误** | 从A一定会导致极端的Z | "用了低代码就会失去工程能力" |
| **稻草人谬误** | 攻击一个没人真正持有的弱化立场 | "反对微服务就是主张单体垄断" |
| **诉诸权威** | 权威说了，所以是对的 | "Google都这么做，所以你也该这么做" |
| **循环论证** | 结论藏在前提里 | "这个系统可靠，因为它被设计为可靠的" |
| **以偏概全** | 从有限样本跳到普遍结论 | "我们团队用了效果好，所以所有团队都适用" |
| **假二元对立** | 把多种选项压缩成非此即彼 | "要么全上云，要么落后于时代" |
| **诉诸新奇** | 新的就是好的 | "这是最新技术，当然更好" |

### 三、隐含前提检验

把论证中没有明说但必须为真才能成立的前提找出来：

**检验方法：**
1. 把每个推论步骤写出来
2. 问：这一步成立，需要额外假设什么？
3. 把额外假设写出来
4. 检验这个额外假设是否真的成立

### 四、定义一致性检验

在整篇论证过程中，关键词的定义有没有发生漂移？

- 开头"性能"指的是延迟，结尾"性能"变成了吞吐量
- 开头"可靠性"指的是不宕机，后面变成了数据不丢失
- 开头"成本"是基础设施成本，后面变成了总体拥有成本
