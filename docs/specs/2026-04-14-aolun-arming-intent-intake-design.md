**日期：** 2026-04-14  
**状态：** 已批准  
**目标 skill：** `skills/aolun-arming/SKILL.md`

## 背景与动机

当前 `aolun-arming` 的路由逻辑仅基于输入文本的字符数（≥1500 → fileflow，<1500 → 内存执行）。路由决策不包含用户意图，导致以下问题：

1. agent 假设用户想要"分析"，但用户可能想要"建设方针"
2. agent 假设用户熟悉该领域，但用户可能需要先做田野调查
3. 用户不知道可以选择 sub-agent 并行执行模式
4. 用户无法表达"分析+建设"的混合目标

这与《实践论》的核心相悖——**认识位置决定所需工具**，在搞清楚用户处于认识过程的哪个位置之前，盲目选择路径是理论推导代替实地调查的典型失误。

---

## 哲学基础

### 《实践论》：认识飞跃的定向

认识经历两个飞跃：感性认识 → 理性认识 → 再回到实践验证。

入口问询的作用是**在第一飞跃前定向**：
- 用户处于感性认识阶段（知道有问题，但没搞清楚）→ 需要解剖分析工具
- 用户已有理性认识，需要实践路径 → 需要建设工具（`aolun-build`）
- 用户在感性认识之前（领域不熟）→ 需要先做田野调查（`aolun-ground`）

### 《矛盾论》：找主要矛盾

在复杂意图中找主要矛盾。Q1 和 Q2 是两个真正正交的维度：
- **认识位置矛盾**：已知信息 vs. 决策所需信息
- **实践目标矛盾**：认识矛盾（分析）vs. 实践矛盾（建设）

这两个维度的笛卡尔积构成完整的路由矩阵。

## 设计决策

### 触发位置与时机

```
1. 收到输入文本
2. 判断长度（现有逻辑，保持不变）
   → ≥1500 字符：强制进入 aolun-fileflow（不走问询，直接移交）
   → <1500 字符：进入意图问询
3. 意图问询（2-3 个问题，逐一问，已明确则跳过）
4. 根据问询结果输出路由决策
```

**关键原则：长文本 ≥1500 字符走 fileflow，不走问询。** fileflow 自身有完整的工作流引导。意图问询仅针对短文本内存执行路径。

### 问题数量与跳过规则

**最多3个问题，逐一提问。** 如果用户提示词或上下文已明确某个维度，跳过该问题直接采用推断结果。

跳过的条件：
- 用户明确说"帮我写反驳/评价/拆解" → Q2 自动选 A（分析）
- 用户明确说"帮我制定方案/下一步怎么做" → Q2 自动选 B（建设）
- 用户说"我不太熟悉这个领域" → Q1 自动选 B（不熟悉）
- 用户没有任何 sub-agent 相关信号 → Q3 展示推荐，不强制

## 3个问题详细规格

### Q1：认识位置（对应《实践论》感性认识层）

**问题：**
> 你对这个领域/文本有多熟悉？

**选项：**
- A. 基本了解，可以直接进入分析
- B. 不太熟悉，需要先做一些背景调查（建议先走 `aolun-ground`）

**路由影响：** 选 B 则在所有路径前插入 `aolun-ground`

**跳过条件：** 用户在提示词中明确表达了熟悉度信号

---

### Q2：主要目标（对应《矛盾论》主要矛盾识别）

**问题：**
> 你的主要目标是什么？

**选项：**
- A. 搞清楚"它到底是什么问题"——解剖分析为主
- B. 知道"下一步应该怎么做"——实践方针为主
- C. 两者都要——**先走完分析路径，再基于分析结论走建设路径**（顺序执行，非并行）

**C 选项的含义明确：** dissect → scan → attack → （以 attack 结论为输入）→ build。这是顺序执行，不是并行。

**路由影响：** 见路由决策矩阵

**跳过条件：** 用户提示词明确偏向分析或建设

---

### Q3：执行模式（横切关注点）

**问题：**
> 是否使用 sub-agent 并行执行？（速度更快，适合多步骤分析；需要宿主平台支持 agent dispatch）

**选项：**
- A. 是，启用 sub-agent 并行（推荐用于四层解剖 + 多维扫描的完整路径）
- B. 否，顺序执行

**路由影响：** 选 A 时，`aolun-scan-orchestrator` 内部启用并行 dispatch；选 B 时顺序调用各 scan-* skill。此问题不影响路径选择，只影响执行模式。

**跳过条件：** 用户选择了仅含单步骤的路径（如只走 `aolun-build`，无需 sub-agent）

## 路由决策矩阵

| Q1 | Q2 | 路由路径 |
|----|----|---------|
| A（熟悉） | A（分析） | dissect-concept → scan-orchestrator → attack |
| A（熟悉） | B（建设） | build |
| A（熟悉） | C（两者） | dissect-concept → scan-orchestrator → attack → build |
| B（不熟悉） | A（分析） | ground → dissect-concept → scan-orchestrator → attack |
| B（不熟悉） | B（建设） | ground → build |
| B（不熟悉） | C（两者） | ground → dissect-concept → scan-orchestrator → attack → build |

**Q3 横切所有行：** 凡包含 scan-orchestrator 的路径，Q3=A 启用并行，Q3=B 顺序执行。
